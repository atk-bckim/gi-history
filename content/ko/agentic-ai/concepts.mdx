# Prompt에서 Agentic Workflow까지

조사 기준일: 2026-06-22

AI 실무는 "프롬프트를 잘 쓰는 법"에서 시작했지만 거기서 끝나지 않는다. 개발자에게 중요한 변화는 모델에게 한 번 답을 받는 기술에서, 모델이 일할 수 있는 컨텍스트, 도구, 권한, 평가, 반복 루프를 설계하는 기술로 이동했다는 점이다.

이 페이지는 용어를 따로 외우는 대신 하나의 계층으로 연결한다.

```text
Prompt
-> Context
-> Retrieval/RAG
-> Tools/MCP
-> Agent
-> Skills
-> Harness
-> Loop
-> Subagents
-> Deep Research
```

## 1. Prompt engineering

Prompt engineering은 모델에게 목표, 맥락, 제약, 예시, 출력 형식을 명확히 전달하는 기술이다. OpenAI prompt engineering 문서는 모델이 텍스트, 코드, 구조화된 JSON 등 다양한 출력을 만들 수 있으며, 요청을 명확히 구성하는 전략이 결과 품질에 영향을 준다고 설명한다.

좋은 프롬프트의 기본 구조:

```text
Goal: 무엇을 달성할 것인가
Context: 어떤 코드, 문서, 사용자 상황이 중요한가
Constraints: 하지 말아야 할 것과 지켜야 할 것
Examples: 좋은 입력/출력 예시
Output: 원하는 형식, 길이, 우선순위
Verification: 어떻게 맞았는지 확인할 것인가
```

프롬프트 엔지니어링은 여전히 중요하다. 다만 agentic workflow에서는 프롬프트가 전체 시스템의 작은 부분이 된다.

## 2. Context engineering

Context engineering은 모델이 볼 정보와 보지 말아야 할 정보를 설계하는 일이다. 파일, 문서, 검색 결과, 대화 기록, 도구 설명, 정책, 메모리, 테스트 출력이 모두 컨텍스트가 될 수 있다.

좋은 컨텍스트 엔지니어링은 많이 넣는 것이 아니라 필요한 것을 선별한다.

- 작업 목표와 직접 관련된 파일
- 실패 로그와 테스트 출력
- 현재 아키텍처와 금지된 패턴
- 결정에 필요한 원문 문서
- 오래되었거나 충돌하는 문서의 제외
- 민감 정보와 불필요한 대화 기록의 제거

개발자 관점에서 context engineering은 "AI onboarding"이다. 새 팀원에게 1,000페이지 문서를 던지는 것보다 현재 작업에 필요한 지도, 규칙, 예시를 주는 편이 낫다.

## 3. RAG

RAG(Retrieval-Augmented Generation)는 외부 지식에서 관련 정보를 검색해 prompt를 보강한 뒤 답을 생성하는 패턴이다. 2020년 RAG 논문은 parametric memory(모델 내부 지식)와 non-parametric memory(검색 가능한 외부 지식)를 결합하는 접근을 제시했다. OpenAI retrieval 문서는 vector stores와 semantic search를 사용해 데이터에서 관련 정보를 검색하는 방식을 설명한다.

RAG가 필요한 경우:

- 모델 학습 시점 이후의 정보가 필요할 때
- 조직 내부 문서나 제품 지식이 필요할 때
- 답변 출처와 citation이 중요할 때
- 지식 업데이트를 모델 재학습 없이 하고 싶을 때

RAG의 품질은 retrieval 품질에 좌우된다.

- 문서 chunking이 적절한가
- metadata와 권한이 유지되는가
- query rewriting이 필요한가
- top-k 결과가 충분하지만 과하지 않은가
- 출처가 답변에 연결되는가
- 검색 실패 시 모델이 추측하지 않게 하는가

## 4. MCP

MCP(Model Context Protocol)는 AI 애플리케이션과 외부 데이터/도구를 연결하는 개방형 프로토콜이다. 2025-06-18 MCP specification은 hosts, clients, servers의 구조와 resources, prompts, tools 같은 server features를 정의한다.

RAG와 MCP는 다르다.

| 개념 | 핵심 |
|---|---|
| RAG | 관련 지식을 검색해 답변 컨텍스트로 넣는다. |
| MCP resources | 외부 데이터와 컨텍스트를 표준 방식으로 노출한다. |
| MCP prompts | 재사용 가능한 prompt나 workflow를 제공한다. |
| MCP tools | 모델이 호출할 수 있는 함수를 제공한다. |

MCP의 위험은 강력함에서 온다. Specification은 사용자 동의, 데이터 개인정보, 도구 안전, sampling control을 핵심 신뢰 원칙으로 다룬다. 도구는 임의 코드 실행이나 외부 시스템 변경으로 이어질 수 있으므로 승인과 권한 경계가 필수다.

## 5. Agents

OpenAI Agents SDK 문서는 agent를 계획하고 도구를 호출하며 specialist와 협업하고 상태를 유지해 multi-step work를 수행하는 애플리케이션으로 설명한다.

Agent는 단순 채팅보다 다음이 추가된다.

- 목표를 여러 단계로 나누는 planning
- 파일, 웹, API, 터미널 같은 tools
- 관찰 결과에 따라 다음 행동을 바꾸는 loop
- guardrails와 approvals
- state와 memory
- handoffs나 agents-as-tools 같은 orchestration
- 결과 평가와 재시도

Agent가 필요한 기준은 "모델이 똑똑한가"가 아니다. 작업이 여러 단계이고, 중간 관찰이 다음 행동을 바꾸며, 도구 호출과 검증이 필요하면 agent 구조가 유리하다.

## 6. Agent skills

Agent Skill은 특정 작업을 더 잘 수행하도록 지침, 참고자료, 스크립트, 템플릿을 묶은 재사용 가능한 패키지다. OpenAI Codex skill 문서는 skill이 `SKILL.md`와 optional scripts, references, assets로 구성되며, Codex가 관련 작업에서 필요한 경우 전체 지침을 로드한다고 설명한다.

Skill은 context budget을 절약하는 방식이기도 하다. 모든 세부 절차를 항상 프롬프트에 넣는 대신, 이름과 설명만 노출하고 필요할 때 자세한 내용을 로드한다.

좋은 skill 후보:

- 반복되는 릴리즈 절차
- 보안 리뷰 방법
- 특정 프레임워크 마이그레이션
- PDF, spreadsheet, slide 같은 특수 파일 처리
- 내부 API client 생성
- 테스트/벤치마크 실행과 보고서 작성

## 7. Harness engineering

Harness engineering은 agent가 안정적으로 일할 수 있는 작업 환경을 설계하는 실무다. OpenAI의 harness engineering 글은 사람이 코드를 직접 쓰는 대신 환경, 의도, feedback loops를 설계해 Codex agents가 reliable work를 하게 만드는 경험을 설명한다.

Harness에 포함되는 것:

- repo-local docs와 architecture maps
- AGENTS.md 같은 진입 지침
- 테스트, lint, typecheck, security scanner
- observability: logs, metrics, traces
- 브라우저/앱 제어 도구
- approval gates와 권한 모델
- PR 템플릿과 review agents
- CI가 agent에게 이해 가능한 오류 메시지를 제공하는 구조

핵심은 agent legibility다. agent가 볼 수 없는 지식은 사실상 존재하지 않는다. Slack 대화, 사람 머릿속 결정, 외부 문서는 repo-local 문서나 실행 가능한 검증으로 옮겨야 한다.

## 8. Loop engineering

Loop engineering은 AI 작업이 반복 실행되는 구조를 설계하는 신흥 실무다. 아직 단일 표준 정의가 있는 용어라기보다, agentic development에서 반복적으로 등장하는 운영 패턴을 가리킨다.

좋은 loop에는 다음이 있다.

- trigger: 무엇이 루프를 시작하는가
- state: 무엇을 기억하고 이어받는가
- budget: 시간, 비용, tool calls 한도
- stop condition: 언제 멈추는가
- evaluation: 성공을 어떻게 판단하는가
- escalation: 언제 사람에게 넘기는가
- audit trail: 무엇이 기록되는가

예:

```text
trigger: 새 PR 생성
loop: test -> review -> fix -> rerun
stop: tests pass and no P1 findings
budget: 3 repair attempts
escalation: same failure twice or migration needed
record: PR comment with attempts and remaining risk
```

루프를 설계하지 않으면 agent는 멈춰야 할 때 계속 시도하거나, 더 조사해야 할 때 성급히 완료를 선언한다.

## 9. Subagent-driven development

Subagent-driven development는 큰 작업을 좁은 역할의 agent들에게 나누어 맡기고, main agent나 사람이 결과를 통합하는 방식이다. OpenAI Codex subagents 문서는 복잡하고 병렬성이 높은 작업에서 specialized agents를 병렬로 실행하고 결과를 모을 수 있다고 설명한다. VS Code subagents 문서는 research, code analysis, specialized review, multi-model consensus 같은 사용 시나리오를 제시한다.

좋은 subagent 사용:

- 보안, 성능, 접근성 리뷰를 각각 분리
- 세 가지 아키텍처 대안을 독립적으로 조사
- 대형 코드베이스에서 영역별 영향 분석
- 구현 agent와 review agent의 컨텍스트 분리

나쁜 사용:

- 단순 작업을 불필요하게 쪼개 token과 시간 낭비
- 모든 subagent에게 전체 저장소와 전체 대화를 넘김
- 책임자 없이 결과만 병렬로 쌓음
- 서로 충돌하는 지시를 검증 없이 합침

Subagent의 핵심 가치는 병렬성보다 context isolation이다.

## 10. Deep research

Deep research는 복잡한 질문을 계획하고, 검색하고, 분석하고, 출처가 있는 보고서로 합성하는 agentic research workflow다. OpenAI deep research API 문서는 clarification, prompt rewriting, research 과정처럼 다단계 흐름을 설명한다.

Deep research가 적합한 질문:

- 여러 출처를 비교해야 하는 시장/기술 조사
- 빠르게 변하는 정책, 제품, 경쟁 정보
- 긴 문서와 웹 자료를 함께 검토해야 하는 주제
- citation이 있는 보고서가 필요한 의사결정

부적합한 질문:

- 이미 정답이 있는 짧은 사실 확인
- 실시간 법률/의료/금융 결정을 대신해야 하는 고위험 판단
- 내부 권한 데이터가 필요한데 connector나 data boundary가 준비되지 않은 경우
- 사람이 원문을 검증하지 않을 보고서

Deep research의 산출물은 최종 진실이 아니라 검증 가능한 research artifact다.

## 운영 훈련 워크플로

실무형 수업의 목표는 용어를 외우게 하는 것이 아니라 팀이 실행할 수 있는 산출물을 만들게 하는 것이다. 다음 이름들은 보편 표준이라기보다 훈련용 패턴이지만, agentic work를 점검 가능한 작업으로 바꿔 준다.

### 딥 인터뷰 스크립트

딥 인터뷰 스크립트는 agent 설계 전에 사용한다. 데모가 이미 그럴듯해진 뒤가 아니라, 요청자, 도메인 담당자, 리스크 담당자, 결과 품질을 판정할 사람에게 먼저 묻는다.

- agent가 어떤 결정이나 행동을 지원해야 하는가
- 어떤 데이터 소스, 도구, 계정을 사용할 수 있는가
- 사용하면 안 되거나 오래되어 신뢰하기 어려운 출처는 무엇인가
- 좋은 답변은 어떤 모습인가, 위험한 답변은 어떤 모습인가
- 어떤 실패는 즉시 사람에게 escalation해야 하는가
- 검토를 위해 agent가 어떤 증거를 남겨야 하는가

산출물은 한 페이지 운영 브리프다. 목표, 허용 컨텍스트, 허용 도구, 승인 게이트, 예시, 평가 기준, stop condition, escalation path를 담는다.

### 브레인스토밍 퍼널

브레인스토밍 퍼널은 모든 문제를 완전 자율 agent로 만들려는 흐름을 막는다. 넓게 시작하고 좁게 수렴한다.

1. 후보 workflow를 나열한다: prompt-only, RAG-assisted, tool-assisted, agentic, subagent-supported.
2. 사용자 가치, 실패 비용, 데이터 민감도, 검증 난이도로 각 후보를 평가한다.
3. 실제 예시로 테스트할 수 있는 가장 작은 workflow를 고른다.
4. source access, permission boundary, evaluation method가 없는 기능은 뒤로 미룬다.

이렇게 하면 brainstorming이 운영 절차가 된다. 학습자는 먼저 "agent가 할 수 있는가?"를 묻지 않는다. "검증 가능한 가장 작은 loop가 무엇인가?"를 묻는다.

### 하네스 준비도 스코어카드

Agent가 파일을 쓰거나, 티켓을 열거나, 메시지를 보내거나, 외부 시스템을 변경하기 전에 하네스 준비도 스코어카드를 사용한다. 각 항목을 0 없음, 1 부분적, 2 신뢰 가능으로 점수화한다.

| 영역 | 준비도 질문 |
|---|---|
| Context | repo-local docs, architecture maps, examples가 있는가 |
| Verification | agent가 tests, lint, typecheck, review checks를 실행할 수 있는가 |
| Permissions | 도구가 account, environment, approval policy로 제한되는가 |
| Observability | reviewer가 logs, commands, diffs, traces, citations를 볼 수 있는가 |
| Evaluation | output quality를 판단할 좋은 예시와 나쁜 예시가 있는가 |
| Escalation | agent가 언제 멈추고 사람에게 물어야 하는지 아는가 |
| Rollback | 잘못된 action을 되돌리거나 격리할 방법이 있는가 |

안전에 중요한 항목 중 하나라도 0이면 workflow를 read-only 또는 human-in-the-loop로 유지한다. 약한 harness는 더 강한 prompt로 해결되지 않는다.

### 서브에이전트 위임 브리프

Subagent-driven development는 위임 범위가 좁고 감사 가능할 때 가장 잘 작동한다. specialist를 실행하기 전에 서브에이전트 위임 브리프를 작성한다.

```text
Role: security reviewer, performance analyst, docs researcher, or implementation scout
Question: 이 subagent가 답해야 할 하나의 결정
Inputs: 검사할 정확한 files, logs, docs, sourceRefs
Exclusions: 피해야 할 files, systems, assumptions
Allowed tools: read-only, tests, web, terminal, browser, or none
Output: findings, evidence, confidence, open questions
Acceptance: main agent가 결과를 검증하는 방법
Owner: recommendation을 통합하거나 거절할 책임자
```

Main agent는 subagent output을 그대로 병합하지 않는다. 증거를 확인하고, 충돌을 해결하고, 필요한 검증을 다시 실행한다.

### Loop safeguards와 Ralph Wiggum Loop

Ralph Wiggum Loop는 agent가 얕은 행동을 반복하면서 매번 관찰 결과를 진전으로 오해하는 실패 모드를 부르는 비공식 훈련용 별칭이다. 정식 산업 표준 용어가 아니라 기억하기 쉬운 위험 패턴으로 사용한다.

다음 loop controls로 방지한다.

- 첫 action 전에 최대 시도 횟수를 정한다
- 매 시도 후 새로운 evidence를 요구한다
- 같은 실패가 두 번 나타나면 멈춘다
- 더 많은 코드나 콘텐츠를 바꾸기 전에 hypothesis를 바꾼다
- 완료 선언 전에는 fresh verification을 요구한다
- 무엇을 바꿨고, 무엇을 관찰했고, 어떤 risk가 남았는지 기록한다
- 다음 단계에 권한, 데이터, ownership이 부족하면 escalation한다

Loop는 학습할 때만 유용하다. 같은 command, 같은 edit, 같은 explanation을 새 evidence 없이 반복하면 loop를 멈추고 딥 인터뷰 스크립트나 하네스 준비도 스코어카드로 돌아간다.

## 개발자용 통합 모델

실무에서는 다음 질문 순서가 유용하다.

1. 단일 답변이면 충분한가? 그러면 prompt engineering.
2. 답변 품질이 입력 정보에 좌우되는가? 그러면 context engineering.
3. 외부 지식 검색이 필요한가? 그러면 RAG.
4. 외부 시스템의 행동이 필요한가? 그러면 MCP tools.
5. 여러 단계와 관찰-수정 루프가 필요한가? 그러면 agent.
6. 반복 절차를 재사용해야 하는가? 그러면 Agent Skill.
7. agent가 안정적으로 일할 환경이 부족한가? 그러면 harness engineering.
8. 작업이 계속 돌아가야 하는가? 그러면 loop engineering.
9. 독립 조사/리뷰/구현이 필요한가? 그러면 subagents.
10. 출처 기반 장시간 조사가 필요한가? 그러면 deep research.

## 체크리스트

- 프롬프트에 목표, 제약, 출력, 검증이 있는가
- 컨텍스트가 충분하지만 과하지 않은가
- RAG 검색 결과와 citation이 품질 기준을 만족하는가
- MCP tools에 승인과 권한 경계가 있는가
- Agent가 중단 조건과 실패 보고 방식을 갖는가
- Skills가 반복 절차를 정확히 캡슐화하는가
- Harness가 테스트, 로그, 문서, 정책을 agent에게 legible하게 제공하는가
- Loop가 예산, stop condition, escalation을 포함하는가
- Subagents가 context isolation을 제공하는가
- Deep research 결과를 원문으로 검증했는가
