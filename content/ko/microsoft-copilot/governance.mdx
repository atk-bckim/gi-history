# Microsoft Copilot 계정, 데이터, 거버넌스

조사 기준일: 2026-06-22

Microsoft Copilot을 교육할 때 가장 먼저 정리해야 할 것은 기능 이름이 아니라 계정과 데이터 경계다. 개인 Microsoft 계정으로 쓰는 Copilot, Entra 계정으로 로그인한 Microsoft 365 Copilot Chat, 라이선스가 필요한 Microsoft 365 Copilot, Copilot Studio로 만든 agents는 겉보기에는 모두 "Copilot"이지만 데이터 접근과 보호 조건이 다르다.

## 제품과 계정 경계

| 경험 | 주 사용자 | 로그인 | 기본 grounding | 핵심 거버넌스 포인트 |
|---|---|---|---|---|
| Microsoft Copilot | 개인 사용자 | 개인 Microsoft account(MSA) | 웹 중심 | 개인 서비스 약관과 개인정보 설정을 확인한다. |
| Microsoft 365 Copilot Chat | 업무/교육 사용자 | Microsoft Entra account | 웹 중심, 사용자가 업로드한 파일, 제한된 agents | Entra 로그인 시 enterprise data protection이 적용된다. |
| Microsoft 365 Copilot | 라이선스가 있는 업무 사용자 | Microsoft Entra account | 웹 + Microsoft Graph의 업무 데이터 | 사용자 권한, 민감도 레이블, Purview, 감사, 보존 정책이 중요하다. |
| Copilot Studio agents | 조직이 만든 업무 agents | 조직 설정에 따름 | 지침, 웹, SharePoint, Graph connectors, 외부 데이터 | agent별 권한, 과금, terms/privacy, admin 승인 흐름을 확인한다. |

교육 문장으로 압축하면 이렇다. "개인 Copilot은 개인용, Microsoft 365 Copilot Chat은 Entra 계정 기반 업무용 채팅, Microsoft 365 Copilot은 Graph 기반 업무 데이터까지 추론하는 라이선스 경험"이다.

## Microsoft Graph grounding

Microsoft 365 Copilot은 사용자의 prompt를 처리하면서 Microsoft Graph의 tenant 데이터와 사용자의 작업 컨텍스트를 grounding에 활용할 수 있다. Microsoft Learn은 문서, 이메일, 일정, 채팅, 회의, 연락처 같은 조직 데이터를 사용자의 권한 안에서 응답에 연결한다고 설명한다.

중요한 점은 "Copilot이 조직 전체 데이터를 다 볼 수 있다"가 아니라 "사용자가 볼 권한이 있는 데이터가 Copilot 응답의 후보가 된다"는 것이다. 그래서 Copilot 도입 전의 가장 현실적인 보안 과제는 프롬프트 교육보다 권한 정리다.

점검할 항목:

- SharePoint와 OneDrive의 과도한 공유 링크
- Teams 채널과 shared channel 권한
- 외부 사용자와 게스트 접근
- 민감도 레이블과 DLP 정책
- 오래된 파일의 보존과 삭제 정책
- Graph connectors로 색인된 외부 데이터 권한

Copilot은 기존 권한 모델을 더 눈에 띄게 만든다. 잘못 공유된 파일은 검색으로도 문제였지만, Copilot은 그 내용을 자연어 답변으로 재구성할 수 있어 사용자가 더 쉽게 발견한다.

## Enterprise data protection

Microsoft Learn은 enterprise data protection(EDP)을 Microsoft Product Terms와 Data Protection Addendum 아래에서 Microsoft 365 Copilot과 Microsoft 365 Copilot Chat 사용자 데이터에 적용되는 통제와 약속으로 설명한다. Entra 계정으로 로그인한 Microsoft 365 Copilot Chat 사용자에게 EDP가 제공되며, Microsoft 365 Copilot과 Copilot Chat의 prompts와 responses는 Microsoft 365 상용 서비스와 유사한 계약적 보호를 받는다.

교육에서 반드시 강조할 포인트:

- prompts, responses, Microsoft Graph를 통해 접근한 데이터는 foundation model 학습에 사용되지 않는다고 Microsoft 문서가 설명한다.
- Copilot은 사용자의 기존 권한과 정책을 따른다.
- prompts와 responses는 감사, eDiscovery, Purview 기능 대상이 될 수 있다. 세부 통제는 구독 플랜에 따라 달라진다.
- 웹 검색 queries는 Microsoft 365 서비스 경계와 다른 데이터 처리 관행을 가질 수 있다.
- agents를 사용할 때는 agent의 privacy statement와 terms of use를 확인해야 한다.

## 웹 검색과 업무 데이터의 차이

Microsoft 365 Copilot Chat은 웹 기반 응답을 제공하고, Microsoft 365 Copilot은 웹과 업무 데이터 모두를 활용할 수 있다. 웹 query는 prompt 전체가 그대로 검색엔진에 전송되는 구조가 아니라, 응답 품질 개선을 위해 생성된 짧은 query가 Bing search service로 전송되는 흐름으로 설명된다. 다만 Microsoft 문서는 web queries가 Microsoft 365와 별도 데이터 처리 관행을 갖는다고 명시한다.

따라서 민감한 업무 내용을 prompt에 넣는 행위와 웹 query 처리 방식은 분리해서 설명해야 한다. 사용자는 "EDP가 있으니 어떤 내용이든 입력해도 된다"가 아니라, "업무 계정 보호가 있더라도 최소 필요 정보만 넣고, 웹 grounding과 agents의 데이터 경계를 확인한다"로 이해해야 한다.

## Agents와 Copilot Studio

Microsoft 365 Copilot Chat 사용자는 agents를 사용할 수 있다. FAQ에 따르면 public website나 instructions 기반 declarative agents는 tenant data에 접근하지 않는 형태가 있고, SharePoint나 Graph connector content에 접근하는 agents는 admin 구성과 과금 설정이 필요할 수 있다.

Agent governance 질문:

- 이 agent는 어떤 knowledge source를 사용하나
- SharePoint, Graph connector, 외부 API 접근이 있는가
- 사용자가 설치하는가, 관리자가 할당하는가
- agent의 terms of use와 privacy statement는 무엇인가
- prompt와 response가 audit/eDiscovery 대상인가
- metered consumption이나 Copilot Credits 비용이 발생하는가
- 민감 데이터가 외부 시스템으로 이동하는가

Agent는 "업무 자동화"이므로 일반 chat보다 권한과 감사가 더 중요하다.

## 관리자 준비 순서

Microsoft Copilot 도입 전에는 다음 순서가 실용적이다.

1. 계정 경계 확정: 개인 MSA 사용과 Entra 업무 계정 사용을 명확히 분리한다.
2. 라이선스 분류: Copilot Chat 대상자와 Microsoft 365 Copilot 라이선스 대상자를 구분한다.
3. 권한 청소: SharePoint, OneDrive, Teams, 외부 공유를 정리한다.
4. 정보 보호: 민감도 레이블, DLP, 보존, eDiscovery, audit 설정을 확인한다.
5. 사용자 교육: prompt에 넣어도 되는 데이터, citation 확인, hallucination 검증을 교육한다.
6. Agent 정책: Copilot Studio, agents store, Graph connectors, 외부 데이터 접근을 승인 흐름에 넣는다.
7. 운영 모니터링: usage, feedback, audit, oversharing 발견, 비용을 정기 리뷰한다.

## 사용자 교육용 원칙

업무 사용자는 다음 다섯 문장을 기억하면 된다.

1. 업무 데이터는 개인 계정 Copilot에 넣지 않는다.
2. 업무/학교 계정으로 로그인했는지 확인한다.
3. Copilot은 내가 볼 수 있는 데이터 범위 안에서 답을 만들 수 있다.
4. 출처와 citation을 확인하고 중요한 결정에는 원문을 검토한다.
5. 민감한 데이터는 최소화하고, agents와 웹 검색 경계를 확인한다.

## 실습

같은 질문을 세 환경으로 나누어 비교한다.

1. 개인 Microsoft Copilot: 공개 정보 중심 질문
2. Microsoft 365 Copilot Chat: 업무 계정으로 웹 기반 질문과 파일 업로드 질문
3. Microsoft 365 Copilot: 권한이 있는 업무 문서, 일정, 메일 맥락을 포함한 질문

비교 항목:

- 어떤 계정으로 로그인했는가
- 어떤 데이터가 응답 근거가 되었는가
- citation이나 출처가 제공되었는가
- prompt/response가 조직 감사 대상인가
- 민감 데이터가 불필요하게 포함되었는가

## 업무 rollout artifacts

업무 교육은 슬라이드 이해에서 끝나면 안 된다. 아래 artifacts는 Microsoft Learn의 계정, Graph, connectors, Purview, Teams recap, Copilot Studio knowledge source 설명을 실제 부서 교육 산출물로 바꾸기 위한 운영 자료다. 라이선스와 기능 사용 가능 여부는 항상 tenant admin이 현재 Microsoft 문서와 tenant 설정에서 확인한다.

### 부서별 rollout playbook

부서별 rollout playbook은 팀이나 업무 기능별로 한 장씩 작성한다.

| 항목 | 기록할 내용 |
|---|---|
| 부서와 owner | 업무 owner, IT owner, security/compliance reviewer, support contact. |
| 승인된 시나리오 | 회의 요약, 초안 검토, 정책 Q&A, 영업 준비, 프로젝트 상태 요약처럼 3-5개 workflow. |
| 승인된 Copilot surface | 개인 Copilot은 업무 데이터 금지, Copilot Chat, Microsoft 365 Copilot, Teams meeting Copilot, 승인된 agent를 명시한다. |
| 데이터 경계 | 어떤 SharePoint site, Teams channel, mailbox, meeting, connector 데이터가 grounding에 쓰일 수 있는지. |
| 위험 통제 | 권한 정리, 민감도 레이블, DLP, retention, audit, 사용자 검증 단계. |
| 교육 증거 | 수료 명단, prompt 실습 결과, citation 확인 샘플, feedback theme. |
| Go/no-go 신호 | 권한 검토 완료, support path 준비, Purview evidence 확보, owner의 residual risk 수용. |

먼저 좁은 cohort에서 playbook을 실행한다. 사용자가 어떤 계정으로 로그인했는지, 어떤 데이터가 응답 근거가 될 수 있는지, 출처를 어떻게 검증하는지 설명할 수 있을 때만 범위를 넓힌다.

### 회의 요약 프롬프트

Teams recap과 Teams의 Microsoft 365 Copilot은 admin policy와 meeting 설정이 허용할 때 논의, 결정, 후속 업무를 요약하는 데 쓸 수 있다. 재사용 가능한 회의 요약 프롬프트를 교육하고, 사용자가 summary를 보내기 전에 transcript, recording, chat, linked files를 확인하도록 한다.

```text
회의 요약 프롬프트:
이 회의를 업무 후속 조치용으로 요약해줘.

내가 접근할 수 있는 meeting transcript, meeting chat, linked files만 사용해.
다음 형식으로 반환해:
1. 결정된 사항
2. owner, due date, dependency가 있는 action items
3. 열린 질문
4. 위험 또는 의견 불일치
5. 중요한 claim마다 source citation 또는 transcript moment

불명확하거나 누락되었거나 transcript coverage가 불완전한 내용은 표시해.
owner, due date, commitment, policy conclusion을 만들어내지 마.
```

실습은 단순하게 운영한다. 두 명이 같은 prompt를 실행하고, action item 목록을 meeting record와 비교한 뒤 출처가 없는 항목을 수정한다.

### connector 결정 등록부

Microsoft 365 Copilot connectors는 외부 업무 데이터를 Copilot 경험으로 가져올 수 있다. Microsoft Learn은 외부 content를 Microsoft Graph에 ingest/index하는 synced connectors와, 데이터를 Microsoft Graph에 index하지 않고 MCP로 query-time retrieval하는 federated connectors를 구분한다. connector 결정 등록부는 "시스템을 연결하자"가 통제 없는 데이터 확장으로 바뀌지 않게 만든다.

| 결정 항목 | 검토 질문 |
|---|---|
| Source system | 어떤 업무 system이나 repository를 노출하는가 |
| Connector model | Synced connector, federated connector, 기존 Graph connector, 또는 연결하지 않음 중 무엇인가 |
| 업무 목적 | 어떤 부서 시나리오에 이 데이터가 필요한가 |
| 권한 source | 어떤 identity, group, ACL, entitlement system이 사용자별 접근 범위를 결정하는가 |
| 데이터 분류 | Public, internal, confidential, regulated, excluded 중 어디에 해당하는가 |
| Retrieval scope | 어떤 object, field, folder, record를 포함하고 제외하는가 |
| Citation behavior | 사용자가 cited source를 열어 답변을 확인할 수 있는가 |
| Owner와 review date | drift, schema change, stale content, retirement를 누가 책임지는가 |

등록부에는 기본 거절 경로를 둔다. 권한 mapping, classification, citation behavior가 불명확하면 connector rollout을 승인하지 않는다.

### Purview 증거 체크리스트

각 rollout wave 전후에는 Purview 증거 체크리스트를 사용한다. 목적은 거버넌스 통제가 그림에 존재한다는 말이 아니라, tenant에서 작동한다는 증거를 남기는 것이다.

- Audit이 켜져 있고 sample Copilot interaction에 user, time, app, accessed resource detail이 남는다.
- prompts와 responses를 조직의 subscription과 policy scope에 맞는 Purview audit, eDiscovery, activity workflow에서 찾을 수 있다.
- Copilot output의 근거가 된 source file의 sensitivity label이 review evidence에 나타난다.
- Copilot interaction history의 retention/deletion 기대값을 사용자와 admin에게 설명한다.
- DLP와 information protection 정책을 permitted example과 blocked example로 각각 테스트한다.
- 과도하게 공유된 SharePoint, OneDrive, Teams, connector content 발견 사항을 owner에게 라우팅한다.
- support team이 compliance, security, legal, records management로 escalation해야 하는 조건을 안다.

이 체크리스트를 보편적인 license promise로 설명하지 않는다. Admin이 tenant에서 활성화된 Microsoft Purview capability에 맞춰 매핑하는 evidence plan으로 다룬다.

### agent knowledge-source 검토

Copilot Studio와 Microsoft 365 declarative agents는 instructions, actions, publication settings, knowledge sources에 의해 통제된다. Microsoft Learn은 public websites, uploaded documents, SharePoint, Dataverse, connectors 기반 enterprise data 같은 knowledge sources를 설명한다. 공개 전과 주요 source 변경 후에는 agent knowledge-source 검토를 수행한다.

| 검토 항목 | 승인 질문 |
|---|---|
| Agent purpose | 이 agent가 어떤 업무 결정이나 workflow를 지원할 수 있는가 |
| Knowledge sources | 어떤 websites, files, SharePoint locations, Dataverse tables, connectors가 답변 근거가 되는가 |
| User authentication | 질문한 사용자가 접근할 수 있는 content만 surfacing되는가 |
| Web grounding | 이 agent에서 broad web search가 disabled, limited, approved 중 무엇인가 |
| Ungrounded answers | model knowledge만으로 답해도 되는가, 그 설정이 허용 가능한가 |
| Source quality | 문서가 최신이고 authoritative하며 hidden instructions나 오래된 policy가 없는가 |
| Terms and privacy | terms of use, privacy statement, owner, support path를 admin과 user가 볼 수 있는가 |
| Test prompts | correct refusal, correct citation, cross-boundary data exposure 방지가 테스트로 증명되는가 |

이 검토는 Microsoft Graph grounding 전 권한 정리의 agent 버전이다. Source가 오래되었거나 과도하게 공유되었거나 애매하면 agent가 그 문제를 반복하기 쉽게 만든다.

## 사용자/admin 운영 cadence

Copilot governance는 launch 이후에도 반복 리듬이 필요하다.

| 주기 | 사용자 교육 활동 | Admin 활동 |
|---|---|---|
| Week 0 | account boundary 교육과 첫 부서별 rollout playbook 실습. | 권한 정리, connector 결정 등록부 검토, Teams meeting policy 확인, Purview evidence baseline. |
| Weeks 1-4 | 회의 요약 프롬프트 예시, citation 확인, 안전한 prompt rewrite로 weekly office hours 운영. | audit signal, oversharing report, connector drift, support ticket, blocked scenario 검토. |
| Monthly | 부서 champion이 유용한 prompts 2개와 수정한 Copilot 오류 1개를 제출. | agent knowledge-source 검토 기록, connector decisions, policy changes, adoption risks 검토. |
| Quarterly | 새 시나리오 교육을 갱신하고 위험 실습을 반복. | Purview 증거 체크리스트 재테스트, 미사용 agents/connectors 정리, product change 반영. |

사용자에게 줄 운영 메시지는 "올바른 계정을 고르고, 필요한 최소 데이터만 쓰고, 출처를 확인하고, 이상한 결과를 보고한다"이다. Admin에게 줄 운영 메시지는 "permissions, connectors, agents, Purview evidence, support ownership을 계속 리뷰한다"이다.

## 체크리스트

- 개인 Copilot과 업무용 Copilot Chat을 명확히 구분했는가
- Entra 계정 로그인과 EDP 적용 조건을 교육했는가
- Graph grounding 전 권한 정리를 완료했는가
- web query 데이터 처리 방식을 별도로 설명했는가
- agents의 데이터 접근과 비용을 관리자 승인 대상으로 두었는가
- Purview, audit, eDiscovery, retention 정책을 확인했는가
- 사용자가 citation과 원문 검증을 습관화했는가
- 각 부서별 rollout playbook이 실제 workflow, owner, evidence와 연결되어 있는가
- 회의 요약 프롬프트 실습을 transcript, chat, source file과 대조했는가
- connector 결정 등록부에 permission mapping과 review date가 기록되어 있는가
- 각 rollout wave마다 Purview 증거 체크리스트를 완료했는가
- 공개된 모든 agent에 agent knowledge-source 검토가 있는가
