# VS Code에서 GitHub Copilot 사용하기

조사 기준일: 2026-06-22

VS Code의 Copilot은 이제 단순 자동완성 도구가 아니다. 같은 편집기 안에서 인라인 제안, 인라인 채팅, Chat view, Agents window, edit mode, agent mode, prompt files, custom instructions, MCP 도구가 함께 동작한다. 개발자는 "프롬프트를 잘 쓰는 사람"보다 "Copilot이 볼 컨텍스트와 실행 경계를 설계하는 사람"에 가까워진다.

이 페이지의 목표는 기능 목록 암기가 아니다. 어떤 작업을 어느 Copilot 표면에 맡길지 판단하고, 작업 전에 컨텍스트와 승인 범위를 정리하는 습관을 만드는 것이다.

## 표면별 역할

| 표면 | 적합한 작업 | 주의점 |
|---|---|---|
| Inline suggestions | 짧은 함수, 반복 코드, 테스트 케이스의 다음 줄 | 주변 코드만 보고 과감히 제안하므로 도메인 규칙은 명시적으로 검토한다. |
| Inline chat | 현재 선택 영역의 빠른 수정, 설명, 리팩터링 | 작은 범위가 좋다. 여러 파일 변경은 Chat view나 agent mode가 낫다. |
| Chat view | 코드 질문, 파일 참조, 단계별 구현 상담 | 필요한 파일과 기대 결과를 직접 지정해야 답변 품질이 올라간다. |
| Edit mode | 여러 파일에 걸친 편집 제안과 리뷰 가능한 변경 | 변경 diff를 읽고 테스트로 검증한 뒤 수락한다. |
| Agent mode | 계획, 파일 수정, 명령 실행, 테스트 재시도까지 포함한 다단계 작업 | 도구 승인, 샌드박스, 민감 파일 접근을 의식해야 한다. |
| Agents window | 여러 프로젝트나 장기 작업을 agent-first로 조율 | 작업 목표와 완료 조건을 더 엄격히 써야 한다. |

가장 흔한 실수는 agent mode에 "알아서 고쳐줘"라고만 쓰는 것이다. Copilot이 도구를 호출할 수 있는 표면에서는 요청이 곧 실행 계획의 입력이 된다. 좋은 요청에는 목표, 범위, 금지 사항, 검증 명령, 완료 기준이 들어간다.

## 표면 선택 매트릭스

표면 선택 매트릭스는 "어느 Copilot 기능을 쓰지?"라는 질문을 취향이 아니라 운영 판단으로 바꿔 준다. 먼저 일을 끝낼 수 있는 가장 좁은 표면에서 시작하고, 더 넓은 컨텍스트, 도구 사용, 반복 검증이 필요할 때만 자율성이 큰 표면으로 올린다.

| 작업 형태 | 시작 표면 | 전환할 표면 | 팀 운영 기준 |
|---|---|---|---|
| 다음 줄이나 테스트 assertion을 완성 | Inline suggestions | Inline chat | 주변 로직을 읽고 나서만 수락한다. |
| 선택한 함수 설명 또는 작은 수정 | Inline chat | Chat view | 한 번에 리뷰할 수 있는 작은 범위로 유지한다. |
| 파일 비교, 버그 추적, 아키텍처 질문 | Chat view | Edit mode | 파일, 심볼, 터미널 출력, 기대 결과를 함께 첨부한다. |
| 이미 알고 있는 여러 파일 수정 | Edit mode | Agent mode | 생성된 diff를 읽은 뒤 수락한다. |
| 조사, 편집, 명령 실행, 테스트 재시도 | Agent mode | Agents window | 계획, 도구 경계, 검증 명령을 먼저 요구한다. |
| 여러 프로젝트나 긴 작업 조율 | Agents window | 별도 task 또는 issue | 완료 기준과 롤백 기준을 명확히 쓴다. |

팀에서는 이 매트릭스를 온보딩 문서나 prompt file 가까이에 두는 것이 좋다. 숙련자가 작은 수정에 agent mode를 남용하는 일을 줄이고, 초보자가 여러 파일 작업을 너무 느슨하게 요청하는 일도 줄인다.

## 컨텍스트 지정의 기본

Copilot Chat은 사용자가 제공한 프롬프트와 현재 작업 환경의 컨텍스트를 조합한다. VS Code 문서는 파일, 선택 영역, 코드베이스, 터미널 출력, 이미지, 도구를 컨텍스트로 추가하는 방식을 계속 확장하고 있다.

실무에서는 다음 순서가 안정적이다.

1. 먼저 작업 단위를 좁힌다. "로그인 전체를 개선"보다 "로그인 실패 시 에러 메시지가 중복 렌더링되는 문제를 수정"이 낫다.
2. 관련 파일을 명시한다. 파일명, 심볼, 선택 영역, 최근 오류 로그를 넣는다.
3. 참조해야 할 규칙을 지정한다. 예: "이 프로젝트의 React Server Components 경계를 유지", "기존 Vitest 패턴을 따름".
4. 검증 방법을 쓴다. 예: "`npm test -- auth`와 `npm run typecheck`가 통과해야 함".
5. 실행 권한을 정한다. 예: "패키지 설치 금지", "마이그레이션 파일 생성 금지", "테스트 명령은 실행해도 됨".

## 컨텍스트 팩

컨텍스트 팩은 같은 변경을 맡은 팀원이 먼저 알아야 할 정보를 작게 묶은 것이다. 모든 파일을 던지는 방식이 아니다. 선택한 Copilot 표면이 일을 시작하는 데 필요한 최소한의 유용한 컨텍스트만 담는다.

Inline suggestions를 넘어서는 작업에는 다음 형태를 사용한다.

```md
목표:
로그인 인증 실패 시 에러 메시지가 중복 렌더링되는 문제를 수정한다.

읽을 파일:
- app/auth/login-form.tsx
- lib/auth/errors.ts
- tests/auth/login-form.test.tsx

규칙:
- 기존 React Server Components 경계를 유지한다.
- 공개 error-code 이름은 바꾸지 않는다.
- 현재 Vitest와 Testing Library 스타일을 따른다.

근거:
- 실패 테스트: `npm test -- auth/login-form`
- 관찰된 UI: "Invalid credentials"가 두 번 표시된다.

허용 작업:
- 나열한 파일과 인접 테스트를 수정한다.
- 테스트와 타입체크 명령을 실행한다.

금지 작업:
- 패키지를 설치하지 않는다.
- 마이그레이션 파일을 만들지 않는다.
- 관련 없는 auth provider를 건드리지 않는다.

완료 기준:
- 중복 메시지가 사라진다.
- 집중 테스트와 타입체크가 통과한다.
- 일반 리뷰가 가능한 작은 diff로 끝난다.
```

같은 흐름이 반복되면 컨텍스트 팩을 prompt file로 저장할 수 있고, 오래 유지되는 규칙은 custom instructions로 승격할 수 있다. 현재 실패 명령처럼 일회성 사실은 정책 파일에 넣지 말고 해당 Chat 요청 안에 남겨 두어야 stale한 지침이 되지 않는다.

## Custom instructions

Custom instructions는 반복해서 말해야 하는 팀 규칙을 파일로 저장하는 방식이다. VS Code는 프로젝트 전체에 항상 적용되는 `.github/copilot-instructions.md`, 파일 패턴에 따라 적용되는 `*.instructions.md`, 조직 수준 지침, 사용자 프로필 지침을 지원한다.

좋은 instructions 파일은 짧고 검증 가능하다.

```md
---
applyTo: "**/*.ts,**/*.tsx"
---

# TypeScript rules

- Preserve existing public APIs unless the task explicitly asks for a breaking change.
- Parse untrusted input at boundaries with the existing validation helper.
- Add or update focused tests for changed behavior.
- Do not introduce new state libraries without an architecture note.
```

나쁜 instructions 파일은 "좋은 코드를 작성하라", "보안에 주의하라"처럼 당연하지만 실행 기준이 없는 문장으로 길어진다. Copilot은 긴 정책 문서를 법무팀처럼 해석하지 않는다. 짧은 명령형 규칙과 실제 코드 예시가 더 잘 작동한다.

## Prompt files

Prompt files는 반복 가능한 단일 작업을 `/` 명령처럼 호출할 수 있게 만든 `.prompt.md` 파일이다. VS Code 기준 workspace prompt files의 기본 위치는 `.github/prompts`이고, 사용자 프로필 수준 prompt files도 만들 수 있다.

Prompt files는 custom instructions와 다르다. Custom instructions는 항상 또는 조건부로 배경 규칙을 제공한다. Prompt files는 "이 작업을 이렇게 실행하라"는 재사용 가능한 작업 템플릿이다.

예를 들어 코드 리뷰 prompt file은 다음처럼 구성할 수 있다.

```md
---
name: review-risky-change
description: "Review a diff for correctness, security, and test gaps."
agent: "agent"
tools: ["search/codebase"]
---

Review the selected change as a senior maintainer.

Focus on:
- correctness regressions
- missing boundary tests
- unsafe data access
- unnecessary public API changes

Return findings first, ordered by severity. Include file paths and concrete fixes.
```

Prompt files는 가벼운 단일 작업에 적합하다. 지속적인 역할, 도구 제한, handoff가 필요한 경우에는 custom agent가 낫고, 스크립트와 참고자료를 함께 배포해야 하면 Agent Skill이 낫다.

## MCP와 도구 컨텍스트

MCP(Model Context Protocol)는 Copilot이 외부 시스템의 데이터와 도구를 표준 방식으로 사용할 수 있게 해 준다. GitHub Docs는 MCP를 IDE, Copilot CLI, GitHub.com agent 흐름 전반에서 Copilot 확장 수단으로 설명한다.

MCP는 "더 많은 컨텍스트"가 아니라 "권한 있는 연결"이다. 데이터베이스, 이슈 트래커, 문서 저장소, 브라우저, 내부 API를 연결하면 Copilot의 능력은 커지지만, 노출될 수 있는 데이터와 실행 가능한 작업도 함께 늘어난다. 따라서 MCP 서버를 추가할 때는 다음을 확인한다.

- 어떤 리소스와 도구가 노출되는가
- 읽기 전용인지 쓰기 권한이 있는가
- 인증 토큰은 어디에 저장되는가
- 도구 호출 전에 사용자가 승인할 수 있는가
- 로그에 프롬프트, 코드, 비밀값이 남는가
- 팀원이 같은 설정을 재현할 수 있는가

## Agent mode 안전 루프

Agent mode는 편집, 명령 실행, 테스트 재시도 같은 다단계 작업에 유용하다. 하지만 AI 출력은 항상 검토 대상이다. VS Code의 trust and safety 문서는 workspace trust, MCP 서버 신뢰, 네트워크 도메인 신뢰, 터미널 명령 샌드박싱, 승인 규칙 같은 통제 장치를 설명한다.

실무 루프는 다음처럼 잡는다.

1. 계획 요청: "먼저 변경 계획만 작성하고 파일은 수정하지 말라."
2. 범위 승인: 계획에서 수정 파일, 테스트, 위험을 확인한다.
3. 구현 실행: agent mode에 필요한 도구만 허용한다.
4. diff 리뷰: 생성된 코드가 요구사항과 맞는지 사람이 읽는다.
5. 검증 실행: 테스트, 타입체크, 린트, 앱 동작 확인을 수행한다.
6. 회고 저장: 반복되는 실수는 instructions, prompt file, test, lint로 승격한다.

## agent-mode dry run

agent-mode dry run은 파일 편집이나 부작용이 있는 도구를 허용하기 전에 진행하는 리허설이다. Agent는 컨텍스트 팩을 읽고 필요한 범위에서 작업공간을 살핀 뒤, 계획, 위험 목록, 수정 예상 파일, 검증 명령을 반환한다. 아직 파일을 수정해서는 안 된다.

예시 요청은 다음과 같다.

```md
Agent mode를 dry run으로만 사용한다.

목표: 로그인 에러 중복 렌더링을 수정한다.
컨텍스트 팩: 위에 나열한 파일과 규칙을 사용한다.

파일을 수정하거나, 설치 명령을 실행하거나, 브랜치를 만들거나, 외부 서비스를 호출하지 말라.
지정된 파일을 읽고 실행할 명령을 제안하는 것은 가능하다.

반환할 내용:
1. 가능성이 높은 원인
2. 수정할 파일 목록
3. 정확한 검증 명령
4. 위험과 롤백 메모
5. 컨텍스트 팩에서 빠진 정보
```

이 dry run은 작업 계획을 리뷰하듯 검토한다. 예상 밖의 파일을 많이 언급하거나, 넓은 리팩터링을 제안하거나, 불필요한 네트워크 접근을 요구하면 구현 전에 요청을 좁힌다. 좋은 dry run은 코드 리뷰를 대체하지 않는다. 나쁜 범위가 diff가 되기 전에 잡아내는 장치다.

## workspace trust 체크리스트

MCP 서버, 생성 코드, 낯선 스크립트가 있는 저장소에서 더 큰 자율성을 허용하기 전에는 이 workspace trust 체크리스트를 사용한다.

- Workspace: 폴더가 신뢰된 작업공간이며, 기대한 저장소에서 온 것이고, 수상한 시작 스크립트가 없는가
- Instructions: 어떤 custom instructions, prompt files, agent skills가 로드되며 현재 코드와 맞는가
- MCP: 어떤 서버가 켜져 있고, 어떤 데이터를 노출하며, 쓰기 권한이 있는가
- Tools: 실제로 필요한 터미널 명령, 파일 편집, 브라우저나 fetch 도구, 외부 서비스는 무엇인가
- Network: 허용 도메인이 작업에 필요한 범위로 제한되어 있으며, 해당 도메인이 대신 쓰기 작업을 수행할 수 있는가
- Sandbox: 가능한 경우 터미널 명령이 sandbox 안에서 실행되며, sandbox가 모든 내장 파일 작업을 덮지는 않는다는 점을 이해했는가
- Secrets: 프롬프트, 로그, 터미널 출력, MCP 응답에 토큰, 비공개 코드, 고객 데이터, 자격 증명이 노출될 가능성이 있는가
- Review: 생성된 모든 변경을 diff view에서 확인하고 commit 전에 검증할 것인가
- Rollback: Agent가 범위를 넘었을 때 되돌릴 checkpoint, 깨끗한 branch, 수동 롤백 계획이 있는가

목적은 Copilot 사용을 막는 것이 아니다. 도구 승인을 습관적 클릭이 아니라 의식적인 엔지니어링 판단으로 만드는 것이다.

## 실전 실행 예시

작은 버그 수정 하나를 골라 세 가지 방식으로 수행한다.

1. Inline chat으로 선택 영역만 고친다.
2. Chat view에서 관련 파일과 테스트 명령을 포함해 요청한다.
3. Agent mode에서 "계획 먼저, 승인 후 구현"으로 진행한다.

비교할 항목은 응답 품질, 변경 범위, 검증 용이성, 사람이 개입해야 하는 지점이다. 결론은 대개 간단하다. 작은 수정은 좁은 표면이 빠르고, 여러 파일과 테스트 루프가 필요한 작업은 agent mode가 강하다. 단, agent mode는 컨텍스트와 권한 설계가 부실하면 가장 빠르게 잘못된 변경을 만든다.

이제 이 실습을 팀 버그에 대한 실전 실행 예시로 바꿔 보자.

시나리오: 결제 화면에서 사용자가 Enter를 누르고 버튼을 빠르게 클릭하면 submit이 두 번 발생한다.

1. 표면 선택: 먼저 Chat view에서 이벤트 흐름을 이해한다. 컴포넌트, hook, 테스트를 함께 고쳐야 한다고 판단될 때만 agent mode로 전환한다.
2. 컨텍스트 팩: 결제 컴포넌트, submit hook, 기존 테스트, 짧은 재현 절차, 기대하는 단일 submit 동작, 집중 테스트 명령을 포함한다.
3. Dry run: agent mode에 파일 수정 없는 계획을 요청한다. 결제 흐름 전체를 다시 쓰거나 테스트를 생략하는 계획은 거절한다.
4. 구현: 이름을 적은 컴포넌트, hook, 테스트 파일만 수정하도록 허용한다. 패키지 설치와 네트워크 접근은 끈다.
5. 검증: 집중 테스트를 먼저 실행하고, 그 다음 결제 관련 테스트 그룹을 실행한다. diff에서 접근성과 disabled 상태 동작을 직접 확인한다.
6. 팀 학습: 같은 패턴이 반복될 가능성이 있으면 "Form submit handler는 진행 중인 중복 제출을 막아야 한다" 같은 짧은 instruction과 회귀 테스트 템플릿을 추가한다.

이 실행 예시는 컨텍스트, 권한, 검증이 서로 다른 문제라는 점을 보여 준다. Copilot은 각 단계를 도울 수 있지만 경계의 소유권은 팀에 남아 있다.

## 체크리스트

- 작업 목표와 완료 기준을 한 문장으로 썼는가
- agent mode로 올리기 전에 표면 선택 매트릭스로 표면을 골랐는가
- 관련 없는 파일을 붙여 넣는 대신 컨텍스트 팩을 준비했는가
- Copilot이 읽어야 할 파일과 읽지 말아야 할 파일을 구분했는가
- custom instructions가 짧고 현재 코드와 일치하는가
- prompt file은 단일 반복 작업으로 제한했는가
- MCP 서버의 데이터 접근과 도구 실행 권한을 확인했는가
- 파일 편집을 승인하기 전에 agent-mode dry run을 진행했는가
- 더 넓은 도구 권한을 주기 전에 workspace trust 체크리스트를 완료했는가
- 생성된 코드를 테스트와 리뷰로 검증했는가
