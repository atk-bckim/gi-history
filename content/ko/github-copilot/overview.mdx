# GitHub Copilot 전체 지도와 학습 순서

조사 기준일: 2026-06-22

GitHub Copilot은 하나의 기능이 아니다. 코드 제안, Chat, edit mode, agent mode, Copilot CLI, cloud/coding agent, code review, custom instructions, prompt files, skills, MCP 연결이 함께 있는 개발 작업 플랫폼에 가깝다.

이 개요는 나머지 강의를 읽기 전의 운영 지도다. 어떤 작업을 어디에서 실행할지, 어떤 컨텍스트가 필요한지, 무엇을 승인해야 하는지, 어떤 검증 증거가 있어야 완료라고 말할 수 있는지 먼저 정리한다.

## 어떤 Copilot 표면을 써야 하는가?

| 작업 유형 | 시작 표면 | 이유 | 검증 습관 |
|---|---|---|---|
| 로컬 함수나 테스트 한 줄 완성 | Inline suggestion | 주변 코드 패턴만으로 충분한 경우가 많다. | 최종 diff를 읽고 focused test를 실행한다. |
| 코드/API 질문 | VS Code Chat | 파일, 선택 영역, 터미널 출력, codebase 컨텍스트를 함께 다룰 수 있다. | 근거 파일을 요구하고 실제 소스와 대조한다. |
| 여러 파일 편집 | VS Code edit mode 또는 agent mode | 편집기 안에서 변경 diff를 리뷰할 수 있다. | 테스트를 요구하고 변경 파일을 모두 읽는다. |
| 터미널/빌드 실패 조사 | Copilot CLI | 로컬 셸, 스크립트, 파일 트리, 명령 출력을 바로 관찰할 수 있다. | 승인 범위를 좁히고 실패 명령을 다시 실행한다. |
| 잘 정의된 이슈 위임 | Cloud/coding agent | GitHub 안에서 비동기로 작업하고 브랜치/PR 형태의 산출물을 만든다. | PR에서 커밋, CI, 로그, 수락 기준을 확인한다. |
| Pull request 사전 리뷰 | Copilot code review | 사람 리뷰 전에 잠재 이슈를 빠르게 표면화한다. | 코멘트는 자동 승인 기준이 아니라 리뷰 입력으로 사용한다. |

핵심 규칙은 단순하다. 좁은 작업은 편집기 가까이에 두고, 길거나 비동기적인 작업은 이슈, 테스트, 리뷰 게이트를 더 강하게 설계한다.

## 플랜, 계정, 정책 경계

Copilot Individual, Copilot Business, Copilot Enterprise는 단순 과금 이름이 아니다. 관리 화면, 조직 정책, 엔터프라이즈 기능, 팀이 표준화할 수 있는 범위가 달라진다.

| 영역 | Individual | Copilot Business | Copilot Enterprise |
|---|---|---|---|
| 주 사용자 | 개인 개발자 | 팀 또는 조직 | 더 깊은 GitHub 통합이 필요한 조직 |
| 관리 통제 | 개인 설정 | 조직 정책, seat 관리, 기능 제어 | 엔터프라이즈 거버넌스와 GitHub.com 확장 경험 |
| 교육 초점 | 개인 생산성 | 공통 워크플로 표준과 보안 기준 | 저장소 간 지식, 거버넌스, 대규모 운영 |
| 교육 리스크 | 개인 습관에 과적합 | 팀 instructions와 리뷰 정책이 필요 | rollout, 감사, 유지보수 루틴이 필요 |

교육을 시작할 때는 먼저 묻는다. 계정의 소유자는 누구인가, 저장소의 소유자는 누구인가, 어떤 조직 정책이 적용되는가, 각 Copilot 표면에 어떤 데이터가 노출될 수 있는가?

## 컨텍스트 모델

Copilot 품질은 컨텍스트에 크게 좌우된다. 개발 작업에서는 프롬프트 문장 하나만으로 충분하지 않은 경우가 많다.

| 컨텍스트 소스 | 적합한 용도 | 실패 모드 |
|---|---|---|
| 열린 파일과 선택 영역 | 작은 수정, 로컬 설명, 리팩터링 | 프로젝트 전체 계약을 놓칠 수 있다. |
| Codebase search | 아키텍처 질문, 심볼 추적 | 요청이 모호하면 관련 없는 파일을 가져올 수 있다. |
| 터미널 출력 | 빌드, 테스트, 런타임 실패 재현 | 로그에 민감 정보가 섞일 수 있다. |
| Prompt files | 반복 가능한 검토된 작업 패턴 | 오래된 prompt file은 낡은 명령을 고정할 수 있다. |
| Custom instructions | 지속적인 팀 표준 | 모호한 instructions는 잡음이 된다. |
| MCP | 외부 시스템, 문서, 이슈, 브라우저, DB, 내부 API | 도구 권한이 학습자의 의도보다 넓을 수 있다. |

MCP는 데이터와 행동을 모두 노출할 수 있기 때문에 특히 강력하다. 모든 MCP 서버는 권한 경계로 본다. 읽기 전용인지, 쓰기 가능한지, 무엇을 로그로 남기는지, 사람 승인이 필요한지 먼저 결정한다.

## 안전한 운영 루프

모든 Copilot 작업은 같은 골격을 따라야 한다.

1. 작업 정의: 목표, 파일, 제외 범위, 사용자에게 보이는 동작.
2. 컨텍스트 부착: 선택 코드, 링크된 이슈, 실패 테스트, 문서, 로그, MCP 소스.
3. 경계 설정: 허용 파일, 금지 명령, 데이터 제한, 승인 게이트.
4. 가장 작은 유용한 변경 실행.
5. 검증: 테스트, typecheck, lint, build, 리뷰 체크리스트, 수동 재현.
6. 학습 축적: 반복되는 기준을 custom instructions, prompt files, tests, skills로 승격.

루프가 중요한 이유는 에이전트가 실제 행동할 수 있기 때문이다. Copilot을 잘 쓰는 사람은 질문만 잘하는 사람이 아니라 답변을 검증 가능하게 만드는 하네스를 설계하는 사람이다.

## 실전 흐름: 버그 리포트에서 PR 리뷰까지

시나리오: 게스트 사용자의 체크아웃 폼에서 검증 메시지가 중복 표시된다.

| 단계 | Copilot 표면 | 요청 형태 | 증거 |
|---|---|---|---|
| 파악 | VS Code Chat | "`Form.tsx`, schema, 실패 테스트를 근거로 validation flow를 설명해줘." | 파일 참조와 의심 원인 |
| 재현 | Copilot CLI | "focused checkout validation test를 실행해. 아직 파일은 수정하지 마." | 실패 명령 출력 |
| 수정 | Agent mode | "checkout form/schema/tests만 수정해. 접근성은 유지하고 focused tests를 실행해." | diff와 테스트 결과 |
| 강화 | Prompt file | 같은 유형을 reusable validation bugfix prompt로 만든다. | 리뷰된 `.prompt.md` |
| 리뷰 | Copilot code review + 사람 리뷰 | Copilot에게 risk comment를 요청하고, 사람 리뷰어가 승인한다. | PR 코멘트, CI, 최종 sign-off |

학습자가 익혀야 할 패턴은 이것이다. 표면을 고르고, 작업을 제한하고, 증거로 검증한 뒤, 반복되는 흐름만 자동화한다.

## 학습 순서

1. VS Code Copilot 기본기부터 시작한다. Inline suggestions, Chat, edit mode, agent mode, context targeting, prompt files, custom instructions, MCP, trust controls를 배운다.
2. Copilot CLI로 이동한다. 터미널 중심 계획, 승인, 명령 관찰, 실패 기반 수정을 연습한다.
3. Cloud/coding agent 위임을 배운다. 수락 기준, 파일 범위, CI 기대치를 가진 이슈형 작업을 작성한다.
4. Code review, security, admin policy를 붙인다. AI 리뷰를 사람 리뷰의 대체가 아니라 보조로 배치한다.
5. Agentic AI 개념을 공부한다. 왜 prompt engineering이 context, tool, harness, loop engineering으로 확장되는지 이해한다.
6. Labs와 showcase로 연습한다. Prompt builder, context stack, agent loop, quiz, 고난도 AI 웹 예제를 직접 조작한다.

## 문제 해결 지도

| 증상 | 흔한 원인 | 해결 |
|---|---|---|
| Copilot이 일반적인 코드만 제안한다 | 파일, 목표, 출력 계약이 부족하다 | 관련 파일을 붙이고 정확한 동작과 테스트를 적는다. |
| Agent가 너무 많은 파일을 바꾼다 | 범위와 금지 경로가 명시되지 않았다 | 허용 경로와 완료 기준을 적어 다시 시작한다. |
| CLI가 위험한 승인을 요구한다 | 작업이 프롬프트보다 넓은 파일/명령 접근을 암시한다 | 거절하고 조사와 구현을 분리한다. |
| Cloud agent PR 품질이 낮다 | 이슈가 너무 크거나 테스트가 없다 | 더 작은 작업, 수락 기준, 검증 명령으로 다시 작성한다. |
| 리뷰 코멘트가 시끄럽다 | 규칙이 테스트나 custom instructions로 고정되지 않았다 | 반복 리뷰 기준을 instructions와 자동 검사로 승격한다. |

## 준비도 체크리스트

- 한 파일 수정, 터미널 실패, GitHub 이슈, PR 리뷰에 각각 어떤 Copilot 표면을 쓸지 설명할 수 있는가?
- 내 계정이 Individual, Business, Enterprise 중 어디에 해당하는지 아는가?
- 저장소 custom instructions가 짧고 최신이며 검증 가능한가?
- 반복 작업은 일회성 배경 설명이 아니라 prompt files로 관리하는가?
- MCP 도구의 데이터 접근, 쓰기 권한, 로그, 승인 흐름을 검토했는가?
- 모든 agentic workflow가 테스트 출력, CI, diff review, 사람 승인 같은 증거로 끝나는가?
